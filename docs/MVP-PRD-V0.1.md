# 冰箱点菜 Web App MVP PRD v0.1

## 1. MVP 范围调整

本版本暂时**不做完整库存管理**。

也就是说，不需要做：

- 食材数量管理
- 食材保质期管理
- 做完后扣库存
- 冰箱长期库存列表
- 库存历史变化

本版本只做一个极简闭环：

> 用户临时勾选今晚有的食材 + 使用已缓存的调料库 + 选择精力值 + 输入个性化要求，AI 生成今晚能做的菜。

---

## 2. MVP 核心功能

### P0 功能

1. 首页
   - 食材入口
   - 调料库入口
   - 精力值选择
   - 用餐人数选择
   - 口味选择
   - 个性化 Prompt 输入框
   - 生成按钮

2. 临时食材勾选
   - 用户本次选择“今晚冰箱里有什么”
   - 不作为长期库存保存
   - 可记录最近使用食材，方便下次快速点选

3. 调料库管理
   - 用户勾选家里已有调料
   - 勾选结果需要本地缓存
   - 下次打开 App 时自动恢复
   - AI 只能使用用户已勾选的调料

4. 精力值选择
   - 快饿死了：5 分钟糊弄
   - 还有口气：15 分钟正常做
   - 精力充沛：折腾一下

5. AI 生成菜谱
   - 根据食材、调料、人数、口味、精力值、用户自由输入要求生成结果
   - 不推荐用户没有的食材和调料
   - 返回结构化菜谱结果

6. 菜谱结果页
   - 展示 1 到 3 个菜谱方案
   - 每个菜谱展示耗时、难度、用到的食材、调料、推荐理由
   - 可查看详情

7. 菜谱详情页
   - 展示完整极简步骤
   - 展示翻车提醒
   - 展示可替换说明
   - 可返回重新生成

### 模块 7：个性化 Prompt 输入框

这是你新增的关键功能。

## 功能定位

给用户一个自由表达入口，用于补充 AI 无法通过标签表达的要求。

例如：

```txt
我今天嗓子不舒服，想吃点热乎清淡的。
```

```txt
不想洗太多锅，最好一锅出。
```

```txt
不要汤，想吃下饭一点的。
```

```txt
家里没有葱姜蒜，别提这些。
```

```txt
想做给小朋友吃，别辣。
```

---

## 输入框设计

标题：

```txt
还有什么特别要求？
```

Placeholder：

```txt
比如：不想洗锅、想吃热乎的、不要辣、最好一锅出……
```

输入类型：

```txt
textarea
```

限制：

```txt
最多 200 字
```

是否必填：

```txt
否
```

---

## 个性化 Prompt 优先级

AI 生成时，用户自由输入的要求应该参与生成，但不能突破硬规则。

优先级建议：

### 最高优先级：硬规则

- 不使用用户没有的食材
- 不使用用户没有的调料
- 遵守饮食禁忌
- 遵守精力值复杂度

### 中优先级：用户自由输入

例如：

- 不想洗锅
- 想吃汤
- 想下饭
- 不想吃鸡蛋
- 想给小孩吃

### 低优先级：普通口味偏好

例如：

- 清淡
- 辣一点
- 家常

如果自由输入和标签冲突，以自由输入为准。

例如：

用户勾选了“辣一点”，但 Prompt 写了“今天胃不舒服，别辣”。

AI 应该按“不辣”处理。

---

### 模块 8：生成按钮

按钮文案推荐：

```txt
用这些凑一顿
```

按钮状态：

未选择食材时：

```txt
先选点食材
```

没有设置调料库时：

```txt
建议先设置调料库
```

但不强制阻断。因为用户可能真的只有盐和油，或者想临时输入。

推荐逻辑：

- 没选食材：禁止生成
- 没选调料：允许生成，但默认只使用“盐、食用油”或提示用户设置
- 已选食材：允许生成

---

## 8.3 步骤复杂度规则

### 快饿死了

最多 3 步。

示例：

```txt
1. 鸡蛋打散，火腿肠切片。
2. 锅里放油，倒入剩米饭、鸡蛋、火腿肠一起炒。
3. 加盐和一点生抽，炒匀就能吃。
```

---

### 还有口气

4 到 6 步。

示例：

```txt
1. 番茄切块，豆腐切小块。
2. 锅里放油，炒番茄到出汁。
3. 加水煮开。
4. 放豆腐煮 3 分钟。
5. 加盐和生抽调味。
6. 出锅。
```

---

### 精力充沛

可以 6 到 8 步，但仍然保持家常。

不建议生成太复杂的饭店菜。

# 11. AI Prompt 规则

## 11.1 System Prompt

```txt
你是一个非常务实的家常菜助手。

你的任务不是推荐精致料理，而是根据用户现在已有的食材、家里已有的调料、用餐人数、口味偏好、精力状态和个性化要求，生成今晚立刻能做的家常菜。

你必须严格遵守：

1. 只能使用用户提供的食材。
2. 只能使用用户提供的调料。
3. 不要推荐需要额外购买的材料。
4. 不要使用高级调料，除非用户明确拥有。
5. 不要假设用户有葱姜蒜，除非调料列表中包含。
6. 菜谱必须简单、家常、可执行。
7. 根据用户精力值调整步骤复杂度。
8. 如果用户有自由输入要求，需要尽量满足。
9. 如果自由输入和标签偏好冲突，以自由输入为准。
10. 但自由输入不能突破“只能使用已有食材和已有调料”的硬规则。
11. 输出必须是 JSON，不要输出 Markdown。
12. 不要输出解释文字，只输出 JSON。
```

---

# 12. 本地缓存设计

## 12.1 需要缓存的数据

### 调料库

必须缓存。

```ts
localStorage.setItem("seasoningLibrary", JSON.stringify(data));
```

用途：

- 下次进入 App 自动恢复
- 避免用户重复勾选
- AI 每次生成时读取该调料库

---

### 当前临时食材

建议缓存。

```ts
localStorage.setItem("selectedIngredients", JSON.stringify(data));
```

用途：

- 用户刷新页面不会丢失
- 从食材页返回首页不会丢失

但它不是库存管理。

---

### 最近使用食材

建议缓存。

```ts
localStorage.setItem("recentIngredients", JSON.stringify(data));
```

用途：

- 下次选择更快
- 首页或食材页展示“最近常用”

---

### 用户偏好

可以缓存。

```ts
localStorage.setItem("cookingPreferences", JSON.stringify(data));
```

包含：

```ts
{
  peopleCount: number;
  energyLevel: string;
  tastePreferences: string[];
}
```

建议缓存这些，因为用户通常习惯稳定。

---

### 个性化 Prompt

不建议长期缓存。

原因：

- 它通常是当次临时需求。
- 比如“今天嗓子不舒服”“不想洗锅”不一定下次还适用。

可以仅在当前页面状态中保存。

如果刷新不想丢失，也可以缓存为：

```ts
currentUserPrompt;
```

但生成完成后可以清空。

# 15. 首页生成按钮校验

点击“用这些凑一顿”时：

## 必须校验

```txt
是否选择了至少 1 个食材？
```

如果没有：

```txt
先选点食材吧，不然我真的没法变出饭。
```

## 建议校验

```txt
是否设置了调料库？
```

如果没有：

```txt
你还没设置调料库，我会默认只按“盐、食用油”来推荐。
```

然后给两个按钮：

```txt
去设置调料库
继续生成
```

# 16. 异常与空状态

## 16.1 AI 返回失败

文案：

```txt
这次厨房小助手卡壳了，再试一次。
```

按钮：

```txt
重新生成
```

## 16.2 食材太少

例如用户只选了“鸡蛋”。

AI 可以返回：

```txt
这些食材有点极限，但还能凑。
```

然后生成：

```txt
煎蛋
水煮蛋
鸡蛋汤
```

但只能使用已有调料。

## 16.3 调料太少

如果用户只有：


AI 仍然可以生成，但不要提：

```txt
生抽
蚝油
料酒
葱姜蒜
胡椒粉
```

# 18. MVP 验收标准

## 功能验收

用户可以完成：

- 从首页进入食材选择页
- 勾选本次食材
- 返回首页后看到已选食材
- 进入调料库页
- 勾选调料
- 刷新页面后调料仍然存在
- 选择精力值
- 输入个性化要求
- 点击生成
- 看到 1 到 3 个菜谱
- 进入菜谱详情页查看步骤

---

## AI 结果验收

AI 必须做到：

- 不使用用户没有的食材
- 不使用用户没有的调料
- 能理解用户自由输入要求
- 精力值为 5 分钟时，步骤明显更短
- 精力值为 15 分钟时，步骤正常
- 精力值为精力充沛时，可以稍微丰富
- 不出现“罗勒叶、迷迭香、淡奶油”等用户没有的材料
- 菜谱口吻家常、直接、接地气

# 19. 最终 MVP 定义

这个版本的核心不是“管理冰箱”，而是：

> **今晚临时点选食材，让 AI 结合你家真实调料和你的当前状态，生成马上能做的菜。**

首版真正要打磨的体验只有一个：

```txt
选食材 → 读调料库 → 选精力值 → 写一句要求 → 生成今晚能吃的
```

产品主按钮可以固定为：

```txt
用这些凑一顿
```

产品核心承诺可以写成：

```txt
不多要一个食材，不乱加一个调料。
```
